# Migrating to Timing/Timeline/Arrangement v2 (`0.8.0`)

- Status: Planned for `0.8.0-beta.1`
- Contract: [`../behavior/timeline-arrangement-v2.md`](../behavior/timeline-arrangement-v2.md)

Timing/Timeline/Arrangement v2 gives Timeline and Arrangement a shared,
tempo-mapped `TimingMap` and removes lossy or ambiguous v1 fields. There are no
deprecated aliases; migrate before upgrading. Use `migrateTimelineProject()` and
`migrateArrangementProject()` (§5 of the contract) for existing v1 project data,
and `normalizeTimingMap()`/strict validators for new construction.

## Package-wide renames and removals

| 0.7 API/field | 0.8 API/action |
| --- | --- |
| `TimingMap.offsetMs` | `TimingMap.startPositionMs` |
| `TimelineEvent.durationBeats` | Remove; Timeline supports point events only |
| `TimelineEvent.value` | Remove; fold custom payload into `TimelineEvent.data` |
| `TimelineTrack.type` | Remove; convey semantics through `data` or `id` naming |
| `TimelineEvent.trackId: "global"` | `TimelineEvent.trackId: null` |
| `TimelineQueryOptions` (no global control) | `TimelineQueryOptions.includeGlobalEvents` (default `true`) |
| `sequenceProjectToTimeline()` | Remove; construct a `TimelineProject` explicitly |
| `ArrangementProject.bpm` | `ArrangementProject.timing: TimingMap` |
| `ArrangementProject` (no `durationBeats`) | `ArrangementProject.durationBeats` (required, explicit) |
| `TimelineProject.version: 1` | `TimelineProject.version: 2` |
| `ArrangementProject.version: 1` | `ArrangementProject.version: 2` |
| `TimelineTrack.data`/`TimelineEvent.data: Record<string, unknown>` | `data?: JsonObject` (JSON-compatible only; numeric leaves must be finite) |

## TimingMap

Before:

```ts
const timing: TimingMap = {
  tempos: [{ beat: 0, bpm: 120 }],
  offsetMs: 250,
};
```

After:

```ts
const timing: TimingMap = {
  tempos: [{ beat: 0, bpm: 120 }],
  startPositionMs: 250,
};
```

`startPositionMs` carries the exact same pre-roll meaning as `offsetMs`; this is
a pure rename, not a semantic change. `beatToMs()`/`msToBeat()` signatures are
unchanged.

## Timeline events and tracks

Before:

```ts
const project: TimelineProject = {
  version: 1,
  timing: createTimingMap({ bpm: 120 }),
  tracks: [{ id: "captions", name: "Captions", enabled: true, type: "text" }],
  events: [
    {
      id: "cue-1",
      trackId: "captions",
      beat: 4,
      durationBeats: 2,
      value: 1,
      type: "caption",
      data: { text: "Hello" },
    },
    {
      id: "cue-2",
      trackId: "global",
      beat: 8,
      type: "marker",
    },
  ],
};
```

After:

```ts
const project: TimelineProject = {
  version: 2,
  timing: createTimingMap({ bpm: 120 }),
  durationBeats: 16,
  tracks: [{ id: "captions", name: "Captions", enabled: true, data: { kind: "text" } }],
  events: [
    {
      id: "cue-1",
      trackId: "captions",
      beat: 4,
      type: "caption",
      data: { text: "Hello" },
    },
    {
      id: "cue-2",
      trackId: null,
      beat: 8,
      type: "marker",
    },
  ],
};
```

Notes:

- `durationBeats` moves from implicit (undefined in v1) to an explicit, required
  Project field. Compute it from your event/section extents plus any trailing
  gap you want, or pass an explicit value to `migrateTimelineProject()`.
- A removed `TimelineEvent.durationBeats`/`value` pair that carried real meaning
  (for example, a highlighted range or an intensity value) has no direct v2
  field. Fold it into `data` under a key of your choosing:
  `data: { intensityValue: 1 }`. `migrateTimelineProject()` warns per affected
  event rather than choosing a key for you; unresolved cases require you to pass
  a conversion callback or accept the warning and edit `data` afterward.
- `TimelineTrack.type` has no replacement field. If you relied on it for
  dispatch branching, use `TimelineEvent.type` (now required) or a `data` key on
  the track instead.
- `trackId: "global"` becomes `trackId: null`. This is a safe, meaning-preserving
  rename with no data loss.
- `data` narrows from `Record<string, unknown>` to `JsonObject` with finite
  numeric leaves. This can break existing consumers who stored non-JSON values
  (functions, `Date`, `undefined`, `NaN`/`Infinity`) in `data`; convert those
  values to a JSON-compatible representation before constructing v2 data.

## Query options

Before:

```ts
getEventsInBeatRange(project, 0, 8, { trackIds: ["captions"] });
// trackIds unintentionally excluded global events too, unless the caller
// happened to include the literal "global" sentinel in the list itself —
// there was no dedicated way to keep global events while restricting to
// specific tracks
```

After:

```ts
getEventsInBeatRange(project, 0, 8, {
  trackIds: ["captions"],
  includeGlobalEvents: true, // independent of trackIds; defaults to true
});
```

An invalid range (`fromBeat > toBeat`, or outside `[0, durationBeats]`) now
throws `RangeError` instead of being silently reordered.

## `sequenceProjectToTimeline()` removal

There is no direct replacement. The function encoded a lossy, implied
substitution relationship between `SequenceProject` and `TimelineProject` that is
not part of the v2 contract (Timeline is a sparse cue scheduler, not a
`ChannelSource`, and does not stand in for Sequence playback). Construct a
`TimelineProject` explicitly from your source data:

```ts
const project: TimelineProject = {
  version: 2,
  timing: createTimingMap({ bpm: sequence.bpm }),
  durationBeats: sequence.stepCount / sequence.stepsPerBeat,
  tracks: sequence.tracks.map((track) => ({ id: track.id, name: track.name, enabled: track.enabled })),
  events: sequence.tracks.flatMap((track) =>
    track.steps.flatMap((value, stepIndex) =>
      track.enabled && value > 0
        ? [{ id: `${track.id}-step-${stepIndex}`, trackId: track.id, beat: stepIndex / sequence.stepsPerBeat, type: "step", data: { value } }]
        : [],
    ),
  ),
};
```

`stepsPerBeat` means "steps per beat," so one step spans `1 / stepsPerBeat`
beats and the whole pattern spans `stepCount / stepsPerBeat` beats.

## Arrangement timing

Before:

```ts
const arrangement: ArrangementProject = {
  version: 1,
  bpm: 128,
  patterns: { verse: versePattern },
  sections: [{ id: "s1", patternId: "verse", startBeat: 0, endBeat: 16 }],
};
```

After:

```ts
const arrangement: ArrangementProject = {
  version: 2,
  timing: createTimingMap({ bpm: 128 }),
  durationBeats: 16,
  patterns: { verse: versePattern },
  sections: [{ id: "s1", patternId: "verse", startBeat: 0, endBeat: 16 }],
};
```

- `bpm: 128` becomes one `TempoEvent` at `beat: 0` inside `timing`.
  `migrateArrangementProject()` performs this mapping automatically.
- `durationBeats` has no v1 source. `migrateArrangementProject()` requires an
  explicit option supplying it and returns `{ ok: false }` when the option is
  omitted; it does not infer a value (for example from section extents) on your
  behalf.
- To add a tempo change, add further `TempoEvent`s to `timing.tempos`; section
  and pattern beat placement is unaffected by tempo changes.

## Using the migration functions

```ts
import { migrateTimelineProject, migrateArrangementProject } from "@vixeq/core";

const timelineResult = migrateTimelineProject(v1TimelineProject, {
  durationBeats: 32,
});

if (!timelineResult.ok) {
  // handle timelineResult.errors
} else {
  const project = timelineResult.project; // TimelineProject v2
  // timelineResult.warnings may be non-empty for removed-field conversions
}

const arrangementResult = migrateArrangementProject(v1ArrangementProject, {
  durationBeats: 64,
});
```

`migrateArrangementProject()` expects a valid v1 arrangement. It returns
`ok:false` for an invalid v1 `bpm`, invalid patterns, invalid sections, or a
`durationBeats` option that is shorter than the largest section `endBeat`.

`normalize*()` functions (`normalizeTimingMap()`, `normalizeTimelineProject()`,
and `normalizeArrangement()`) repair data within one schema version and never
change `version`. They are not a substitute for migration across versions.
