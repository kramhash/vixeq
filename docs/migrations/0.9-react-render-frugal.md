# Migrating `latestEvent` to `latestEventRef`

- Status: Unreleased — see `CHANGELOG.md` "Unreleased" → "Breaking"; landing
  pre-1.0 under lockstep versioning, not yet published to a stable version
- Contract: [`../plans/v1-collaboration-spec.md`](../plans/v1-collaboration-spec.md) §4, §7
- Review: [`../reviews/r6-render-frugal-latestref-claude.md`](../reviews/r6-render-frugal-latestref-claude.md)

`useSequencerEngine`/`useSequencePlayer`, `useArrangement`, and `useTimeline`
previously exposed the latest step/cue/playback/project event through a
`useState`-backed `latestEvent` field. That field was updated on every
scheduled step (or cue), which forced **every** consumer of the hook to
re-render on every step — regardless of whether the consumer read the field.
A repository-wide audit found exactly one consumer reading it
(`@vixeq/player-react`'s `SequencePlayer`, to highlight the playing step);
every other consumer either sourced step data from the `onStep`/`onCue`
callback into its own state, or paid the re-render for nothing.

`latestEvent` is now exposed as a ref, mirroring the existing
`positionRef` pattern for continuous position data. This is a breaking
change to the field's shape, not just its name.

## Package-wide rename

| 0.8 API/field | 0.9 API/action |
| --- | --- |
| `SequencerEngineHookState.latestEvent: T \| null` | `latestEventRef: MutableRefObject<T \| null>` |
| `UseArrangementState.latestEvent: T \| null` | `latestEventRef: MutableRefObject<T \| null>` |
| `UseTimelineState.latestEvent: T \| null` | `latestEventRef: MutableRefObject<T \| null>` |
| `SequencePlayerTransportState.latestEvent` | `latestEventRef` (mirrored type, `@vixeq/player-react`) |

## Reading the latest event

Before:

```tsx
function Sequencer({ project }: { project: SequenceProject }) {
  const { latestEvent } = useSequencerEngine({ project });

  useEffect(() => {
    if (latestEvent && "stepIndex" in latestEvent) {
      console.log("step", latestEvent.stepIndex);
    }
  }, [latestEvent]); // re-runs (and re-renders the component) on every step
}
```

After — for occasional/imperative reads (e.g. inside another callback or
effect that already has its own trigger), read the ref directly:

```tsx
function Sequencer({ project }: { project: SequenceProject }) {
  const { latestEventRef } = useSequencerEngine({ project });

  someOtherEffect(() => {
    const event = latestEventRef.current;
    // ...
  });
}
```

After — if you need to repaint on every step (a moving playhead, a live
readout), track `onStep` into your own state instead. This is exactly what
`SequencePlayer` does internally:

```tsx
function Sequencer({ project }: { project: SequenceProject }) {
  const [latestStep, setLatestStep] = useState<StepEvent | null>(null);
  useSequencerEngine({ project, onStep: setLatestStep });

  return <p>Step {latestStep?.stepIndex ?? "–"}</p>;
}
```

A ref mutation does not itself schedule a render — reading `latestEventRef`
inside a render body will not pick up subsequent step events. Use `onStep`/
`onCue` (or `onPlaybackChange`/`onPosition` for their respective events) when
the component must repaint per event.

## `SequencePlayer` is unaffected

`SequencePlayer`'s built-in playhead/step highlight behavior is unchanged —
it migrated to the `onStep`-into-local-state pattern internally, and its
internal `onPlaybackChange` handler resets that local state on `stop()` so
the highlight correctly returns to step 0 (the engine resets its step
position on stop but emits no `"step"` event to drive that reset otherwise).
No action is needed by consumers of `@vixeq/player-react`'s
`<SequencePlayer>` component; this only affects code reading `latestEvent`
directly off the `useSequencerEngine`/`useSequencePlayer`/`useArrangement`/
`useTimeline` hooks, or off `SequencePlayerTransportState` if you built a
custom transport UI against that mirrored type.
