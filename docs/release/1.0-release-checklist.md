# 1.0 Release Checklist

This checklist is the release-readiness gate for `1.0.0-rc.1` and `1.0.0`.
It does not publish by itself; it records the required checks and release order.

## Preconditions

- P0-P8, T0-T8, R0-R6, and RC1A-RC1E are `done` in
  [`../plans/v1-collaboration-spec.md`](../plans/v1-collaboration-spec.md).
- `CHANGELOG.md`, package READMEs, generated API docs, migration guides, and
  [`../../SUPPORT.md`](../../SUPPORT.md) describe the public behavior being
  released.
- Public package versions are bumped in lockstep.
- API Extractor reports are current for all public packages.
- The working tree contains no unrelated release artifacts such as `.tgz`,
  `.tmp/`, copied example output, or generated probe files.
- The Core branch-coverage gap recorded in the 0.9 release checklist is closed
  (`pnpm test:coverage` meets the configured thresholds and is wired into
  `.github/workflows/ci.yml`), or a new approved decision re-scopes the gap.
  `1.0.0-rc.1` must not publish while this gap is open and undecided.

## Required Local Gates

Run these from the repository root before publishing an RC or stable release:

```sh
pnpm typecheck
pnpm build
pnpm test
pnpm test:coverage
pnpm api:check
pnpm behavior:check
pnpm smoke:pack
SMOKE_REACT_VERSION=18 pnpm smoke:pack
SMOKE_TS_VERSION=5.5 pnpm smoke:pack
pnpm test:e2e
```

`pnpm test:coverage` is also wired into CI on the Node 22 build leg once the
RC1 critical-file coverage work is complete; the Node 24 leg continues to run
the ordinary workspace test suite.

## RC Publish Steps

1. Confirm npm authentication and dist-tag state:
   `npm whoami` and `npm view @vixeq/core dist-tags versions --json`.
2. Bump root and public package versions in lockstep to `1.0.0-rc.1`.
3. Re-run the required local gates after the version bump.
4. Publish `@vixeq/core`, `@vixeq/react`, and `@vixeq/player-react` with the
   `next` tag from their package directories:
   `pnpm publish --tag next --access public`.
5. Install the published RC packages in a clean temporary consumer and repeat
   the packed-package smoke coverage against the registry packages.
6. Verify npm metadata for all packages:
   `npm view @vixeq/core@1.0.0-rc.1 version dist-tags exports`.
7. Record package URLs, gate results, known caveats, and migration links in
   the release notes.

## RC Observation Period

Keep the RC public for at least 14 days before stable promotion.

During the observation period:

- Do not make public API changes unless a new RC will be published and the
  observation period restarted.
- Fix blocker or critical defects in a new RC.
- Re-run the required gates for every RC.
- Keep `latest` on the previous stable until stable promotion.

## Stable Promotion Steps

1. Confirm the final RC completed at least 14 days without blocker or critical
   defects and without public API changes.
2. Bump root and public package versions in lockstep to `1.0.0`.
3. Re-run the required local gates.
4. Publish the three public packages with `--tag latest`.
5. Install the published `1.0.0` packages in a clean temporary consumer and
   repeat the registry smoke coverage.
6. Verify npm metadata and dist-tags for all packages.
7. Update `CHANGELOG.md` with package URLs, published dist-tags, gate results,
   and any known support caveats.

## Support Caveats to Record

- Browser support follows the locked Playwright browser set recorded in
  [`../../SUPPORT.md`](../../SUPPORT.md).
- WebKit and CI Firefox browser E2E do not assert real WebAudio playback
  start/progression when Linux headless Playwright cannot start or advance
  `AudioContext.currentTime`.
- Node 20 is not supported for the 1.0 line.
